# Spec: `live-observer-pattern`

## Intent

When writing test scenarios that drive an asynchronous, multi-phase
workflow against a real backend (GitHub, a deployed service, a queue
worker), the natural shape is **one class per scenario family**, holding
per-run state across phase calls and exposing a single `__call__` that
matches the test framework's observe-function signature.

This spec captures the architecture proven out by
`test-harness/lib/live_observe.BatchJobObserver` for the
`batch-job-happy-path` scenario in PR #5. Four more scenarios
(orchestrate-issue-single-subagent, orchestrate-issue-parallel-fanout,
orchestrate-issue-restart-recovery, task-dag-claim-and-plan) are
scheduled to use the same architecture; this spec is what each of them
should follow. Beyond this project, the pattern generalizes to any
"drive-and-observe" test harness for async systems.

## Trigger

### Direct triggers
- "Write a live observer for scenario `<id>`."
- "Add a class that drives `<workflow>` against real GitHub."
- "I need a test fixture that polls the backend until terminal."

### Proactive triggers
- A new scenario YAML with `target: live-new-repo` (or equivalent) is
  added to the harness, and a passing implementation requires phase
  methods that talk to a real backend.
- An existing observer class is being refactored because its phases now
  need cross-call state (issue number, branch SHA, in-flight request id).

### Negative triggers
- The scenario is `target: synthetic-fixture` (no real backend) and the
  phase observations can be computed from on-disk state alone. Use the
  fixture-only observer; the live-observer pattern is overkill there.
- The scenario's phases are independent (no cross-phase state). A
  module-level function per phase is simpler.

## Inputs

- The scenario YAML at
  `test-harness/scenarios/<scenario-id>.yml`, listing phases +
  per-phase `inputs` + `expected` assertion keys.
- An object honouring the project's `GitHubClient` Protocol (the
  in-memory client `InMemoryGitHubClient` for unit tests; `RestGitHubClient`
  for live runs).
- The agent's GitHub login (needed for some envelope fields and for
  matching the workflow's `author_association` gate).
- Optional clock + sleep callables for deterministic polling tests.

## Outputs

- A class in `test-harness/lib/live_observe.py` named
  `<Scenario>Observer` (e.g. `BatchJobObserver`).
- Registration in `live_observe._OBSERVER_FACTORIES` so
  `live_observe.make_observer(scenario_id, ...)` finds it.
- A `live_observer_factory(*, github_client, owner, repo)` function in
  the scenario's runner script that returns an instance.
- A unit-test module at
  `test-harness/tests/test_live_observe_<scenario>.py` (or extending
  the existing `test_live_observe.py`).

## Workflow

1. **Read the scenario YAML.** Identify the phase names and the
   `expected` keys each phase asserts on. These keys are the
   observations the class must return.
2. **Write a `_LiveState` dataclass** with one field per cross-phase
   piece of state (issue number, request comment id, branch, SHA,
   terminal envelope, anything else the verify phase needs from
   earlier phases).
3. **Write the observer class** with `__init__` parameters:
   - `github_client` (positional or keyword) — required.
   - `agent_login` — required; raise `ValueError` if empty.
   - `poll_interval_s` + `poll_timeout_s` — defaults appropriate for
     the workflow's expected handler latency.
   - `sleep`, `clock`, `iso_now` — callables, default to stdlib;
     tests inject substitutes.
   - Any scenario-specific knobs (`request_branch`, `command`,
     `subagent_id`).
4. **Implement `__call__(phase_name, inputs, fixture, diagnostics)`**
   that dispatches by phase name to private `_observe_<phase>` methods.
   Raise `ValueError` on unknown phases.
5. **Per-phase methods do real work AND return observations.** The
   setup phase opens the issue + records sha; the invoke phase posts
   the request + polls; the verify phase reads stored terminal state.
   Each returns a dict whose keys match the scenario's `expected`
   block.
6. **Polling phases use the injected clock + sleep.** No bare
   `time.sleep` or `time.monotonic`. Polling loops have a deadline
   computed from `self._clock() + self._poll_timeout_s`.
7. **Tolerate envelope-body trailing prose.** Use the project's
   lenient envelope parser; never assume `json.loads(body)` will work
   — the GitHub MCP appends a "Generated by Claude Code" footer.
8. **Register the observer in `_OBSERVER_FACTORIES`.**
9. **Wire the runner script** to expose
   `live_observer_factory(*, github_client, owner, repo)` and pass it
   to `cli_main` as the `live_observer_factory` kwarg.
10. **Write unit tests.** At minimum:
    - Construction validation (raises on empty agent_login).
    - Each phase's success path against a scriptable fake client.
    - Each phase's failure-mode path (missing branch, missing prior
      state, polling timeout, trailing-prose tolerance).
    - The unknown-phase path.
    - The factory's `make_observer` entry.

## Concrete examples

### Example 1 — the existing `BatchJobObserver`

```python
# test-harness/lib/live_observe.py (excerpt)

@dataclass
class _LiveState:
    issue_number: Optional[int] = None
    request_comment_id: Optional[int] = None
    request_branch: Optional[str] = None
    request_commit_sha: Optional[str] = None
    terminal_envelope: Optional[dict[str, Any]] = None


class BatchJobObserver:
    def __init__(self, *, github_client, agent_login, ...,
                 poll_interval_s=5.0, poll_timeout_s=300.0,
                 sleep=None, clock=None, iso_now=None):
        if not agent_login:
            raise ValueError("agent_login is required")
        self._client = github_client
        self._sleep = sleep or time.sleep
        self._clock = clock or time.monotonic
        ...
        self.state = _LiveState()

    def __call__(self, phase_name, inputs, fixture, diagnostics):
        if phase_name == "setup":   return self._observe_setup(inputs)
        if phase_name == "invoke":  return self._observe_invoke(inputs)
        if phase_name == "verify":  return self._observe_verify(inputs)
        raise ValueError(f"BatchJobObserver: unknown phase {phase_name!r}")

    def _poll_until_terminal(self, comment_id):
        deadline = self._clock() + float(self._poll_timeout_s)
        while True:
            if self._clock() >= deadline:
                return None
            comment = self._client.get_comment(comment_id)
            parsed = envelopes.parse(comment.get("body"))
            if parsed is not None and envelopes.is_terminal(parsed):
                return parsed
            self._sleep(self._poll_interval_s)
```

The runner side:

```python
# test-harness/runners/batch-job-happy-path.py
def live_observer_factory(*, github_client, owner, repo):
    from live_observe import BatchJobObserver
    return BatchJobObserver(
        github_client=github_client,
        agent_login=os.environ.get("AGENT_LOGIN") or "agent",
    )

cli_main(SCENARIO_ID, observe, live_observer_factory=live_observer_factory)
```

### Example 2 — proposed `OrchestrateIssueObserver` (not yet built)

For `orchestrate-issue-single-subagent`, the phases are `setup` /
`claim` / `fanout` / `merge` / `verify`. The cross-phase state needs:
issue number, feature branch name, claim comment id, subagent
sub-branch SHA, PR number.

Skeleton:

```python
@dataclass
class _OrchestrateState:
    issue_number: Optional[int] = None
    claim_comment_id: Optional[int] = None
    feature_branch: Optional[str] = None
    subagent_branches: list[str] = field(default_factory=list)
    pr_number: Optional[int] = None
    pr_merged: bool = False


class OrchestrateIssueObserver:
    def __init__(self, *, github_client, agent_login,
                 max_parallel=1, poll_interval_s=10.0,
                 poll_timeout_s=600.0, ...):
        if not agent_login: raise ValueError(...)
        ...
        self.state = _OrchestrateState()

    def __call__(self, phase_name, inputs, fixture, diagnostics):
        ...

    def _observe_claim(self, inputs):
        # post claim envelope, wait for issue_locked == True
        ...

    def _observe_fanout(self, inputs):
        # post N subagent dispatch comments, record their ids
        ...

    def _observe_merge(self, inputs):
        # poll for feature branch advance + PR open
        ...

    def _observe_verify(self, inputs):
        # check PR merged + meta_status: shipped on issue body
        ...
```

The pattern is identical; the scenario-specific knowledge lives in the
phase methods.

## Anti-patterns

- **Don't write phase methods as module-level functions** that take all
  cross-phase state via the `diagnostics` dict. The class shape is the
  right granularity — each scenario has its own state shape; one
  class per scenario is cleaner than one giant function table.
- **Don't call `time.sleep` or `time.monotonic` directly.** The whole
  point of injectable clock + sleep is that tests don't actually wait.
  Bare calls re-introduce flakiness.
- **Don't assume `json.loads(comment_body)` works.** Always use the
  project's lenient envelope parser (`envelopes.parse`). MCP-posted
  comments have a trailer.
- **Don't reach into `self._client` directly from tests.** Tests
  construct the client + drive the observer; assertions go against
  what the observer returned + what the client recorded. Reaching past
  the observer couples tests to the client's internal state.
- **Don't make polling-loop deadlines time-of-day-based.** Use the
  injected `clock()` only. Tests with a synthetic clock counter need
  to be able to advance time without `time.sleep`.
- **Don't crash on optional permissions failures.** The setup phase's
  `add_label` call is wrapped in `try / except Exception: pass`
  because the workflow's `lock-and-sweep` applies the label
  regardless. Hard failures on optional steps make the observer
  fragile.
- **Don't skip the unknown-phase guard.** A typo in a scenario YAML
  (`name: invokr`) should raise `ValueError("unknown phase 'invokr'")`,
  not silently no-op.

## Acceptance criteria

1. The class can be constructed with only `github_client` +
   `agent_login`. Defaults for everything else.
2. Each phase method either returns a dict whose keys exactly match
   the scenario YAML's `expected` block, OR raises a meaningful
   exception (not a `KeyError`).
3. With an `InMemoryGitHubClient` + a background fake-handler thread,
   the observer can run all phases end-to-end and the smoke test
   passes.
4. Tests pin polling behaviour (count of `get_comment` calls before
   terminalization) using a scriptable fake's call counter.
5. Tests cover at least: success path, timeout, trailing-prose
   tolerance, missing-prior-state, unknown phase.
6. The runner script's `live_observer_factory` accepts the standard
   `(github_client, owner, repo)` kwargs and constructs the observer
   without any further setup.

## Files this skill creates / modifies

- **Modifies**: `test-harness/lib/live_observe.py` — adds the new
  `<Scenario>Observer` class + `_LiveState`-equivalent dataclass +
  registration in `_OBSERVER_FACTORIES`.
- **Creates / modifies**:
  `test-harness/tests/test_live_observe.py` (or a per-scenario
  variant) with the unit tests.
- **Modifies**: `test-harness/runners/<scenario-id>.py` to expose
  `live_observer_factory` and pass it to `cli_main`.
- **(Optional) Creates**:
  `test-harness/tests/test_runner_<scenario-id>.py` for an end-to-end
  smoke test, following the
  `test_runner_batch_job_happy_path.py` pattern.
