## Live test results

From `runs/Vs1aL/test-results.md`:

- Scenarios driven: 18
- Full pass: 1 (`composition-guide-render`)
- Partial pass (some phases passed synthetically): 4
  (`onboarding-blank-repo`, `onboarding-existing-agents-md`,
  `onboarding-decline`, `protocol-installed-not-onboarded`)
- Fully skipped (every phase needs live skill execution): 13
- Failed: 0

Phase-level: 6/72 phases `done`, 66/72 `skipped`, 0 `failed`.

The 8% phase-pass rate is below the plan's "~80% green" target. The
gap is structural, not symptomatic: most scenarios encode assertions
against artifacts only a live skill+GHA loop can produce (issue
numbers, PR merges, batch-job comment envelopes). The harness honestly
distinguishes "passed synthetically", "skipped (live-required)", and
"failed (real assertion violation)".

## Bugs surfaced

1. **MCP scope has no label-create method.** Both `batch-job` and
   `orchestrate-issue` skill self-install logic claim the `agent-task`
   label gets created via `mcp__github__` (REST POST). The tools
   exposed by the github MCP in this environment include `get_label`
   but not a create-label tool, so the install contract has no path to
   create the label. The skill's `templates/agent/scripts/handler.py`
   could create the label via the REST helper, but that runs server-
   side in GHA, not in the dispatcher.
   **Impact:** Phase 2 self-install is incomplete (label was not
   created); Phase 7 dogfood blocked.
   **Suggested fix:** Either (a) add a `create_label` MCP tool to the
   default github MCP server, or (b) document a manual one-time setup
   step in `bootstrap/install.md` Section 5 (after the initial commit).

2. **MCP scope restricted to current repo, no fresh-repo creation.**
   `test-harness/SPEC.md` and SKILL.md both assume `target: live-new-repo`
   creates `<agent-login>/<run_id>-<scenario_id>` via
   `mcp__github__create_repository`. The repo MCP scope here is
   single-repo (`lago-morph/pipeline-ai-sandbox`), so even though
   `create_repository` is reachable, the agent cannot use it for repos
   outside its scope. Runners degrade to synthetic-fixture and record
   `degraded_reason` in `state.json :: diagnostics`.
   **Impact:** Phase 5 cannot exercise live-only assertions.
   **Suggested fix:** When dispatching to runs that need live scope,
   either (a) widen the MCP scope to include the agent's personal
   account, or (b) materialise live scenarios to a dedicated
   `pipeline-ai-sandbox-harness-runs` repo and add it to scope.

3. **`.gitignore` from the bootstrap matches `lib/` at any depth.**
   The bundled `.gitignore` is a vendored Python template with `lib/`
   on a top-level line, which (per gitignore semantics without a
   leading slash) also matches `test-harness/lib/`. Phase 1 sub-03
   appended pipeline-specific entries but didn't tighten the existing
   `lib/` line, so Phase 4's new `test-harness/lib/` was silently
   ignored on first `git status`. Fixed in this run by tightening
   `lib/` -> `/lib/` and `lib64/` -> `/lib64/`.
   **Suggested fix:** Either (a) fix the upstream `.gitignore` in the
   bundle to use `/lib/` and `/lib64/`, or (b) document the gotcha in
   `bootstrap/install.md` Section 3.

4. **No `test-harness/lib/*.py` modules ship in the bootstrap bundle.**
   `test-harness/SPEC.md` (line 273-279) lists the expected harness
   helpers (`archetype_loader.py`, `scenario_runner.py`,
   `assertions.py`, `state.py`). None of them is in the bundle (only
   the YAML scenario specs and archetype fixtures are). Phase 4 had
   to write the lib from scratch.
   **Suggested fix:** Add the harness helpers to the bundle (POC's
   `pipeline-skills-package/test-harness/lib/`).

5. **Orphan-branch checkout in Phase 2 nearly clobbered Phase 2's own
   work.** `git checkout --orphan _agent_runs && git rm -rf . &&
   git clean -fdx` is what the install procedure looks like in the
   skill SKILL.md, but in this run that sequence ran against an
   already-extracted-but-uncommitted Phase 2 working tree, and
   `git clean -fdx` removed the freshly-copied `.agent/` and three
   workflow files because they were untracked. The dispatcher caught
   it because `git status` was clean after the round-trip; redoing
   the copies fixed it.
   **Suggested fix:** The orphan-branch creation procedure should
   `git stash --include-untracked` (or use a worktree) before doing
   `git clean -fdx`. Best fix is to use `git worktree add --detach`
   and write into the detached worktree, then push the resulting ref.

## Workarounds invented

- **Live-target degradation:** `scenario_runner.run_scenario` recognises
  `target == "live-new-repo"`, records the MCP-scope constraint as a
  `degraded_reason` in state diagnostics, and proceeds as if
  `target=synthetic-fixture`. Honest, restart-safe, restartable under a
  less-restricted dispatcher.
- **Phase 7 dispatcher-less filing:** Issue #1 was opened with a real
  `agent-meta` block but explicit "not driven live" prose; a maintainer
  can apply the `agent-task` label and dispatch the issue when MCP
  scope permits.

## Patterns to harvest

- **Honest synthetic-mode harness.** The split between "synthetic-
  catalogue keys" (`SYNTHETIC_CATALOGUE` in
  `test-harness/lib/synthetic_observe.py`) and "live-required keys"
  produced a runnable test report with zero false positives. This
  pattern - explicitly enumerate what your mock can answer, raise
  `NotImplementedError` for the rest, treat that as `skipped` rather
  than `failed` - is reusable beyond this harness. Worth proposing as
  a reusable convention in `docs/` (or even a skill if the pattern
  generalises).

- **Disjoint-paths parallel fanout without `isolation: "worktree"`.**
  Phase 1's 4 subagents shared the working tree but never collided
  because each held a disjoint write scope. For tasks with clean path
  partitioning, worktree isolation is overhead. The plan should be
  willing to drop it where the partition is obviously safe (and keep
  it where multiple agents might re-touch the same paths).

- **Runner-template generation from scenario YAML.** Writing 18 runner
  files by hand or by 18 subagents would have eaten the run budget.
  Generating them from a template + the scenario YAML was a 30-line
  Python script and produced uniform, restart-safe runners. The
  composition-guide skill could reasonably document "the scenario
  YAML is the source of truth; everything else can be generated."

## Specs / SPEC.md updates needed

These are recommendations for the POC bundle source, not edits to this
repo's `docs/` (which are bootstrap-installed and should round-trip
through the POC):

1. `test-harness/SPEC.md` should list `lib/synthetic_observe.py`
   alongside the other helpers.
2. `test-harness/SPEC.md` should document the `degraded_reason`
   diagnostic and when a runner should set it.
3. `docs/skills/batch-job/SPEC.md` and
   `docs/skills/orchestrate-issue/SPEC.md` should clarify the
   label-create dependency - either name the MCP method explicitly or
   give a `gh api` / `curl` fallback.
4. `bootstrap/install.md` Section 3 should mention the `.gitignore`
   `lib/` gotcha.
5. `bootstrap/install.md` Section 5 should add an optional "Apply
   labels" subsection that runs `gh label create agent-task ...` (or
   the equivalent REST call) if the MCP scope cannot create labels.
