## Part 1 — what happened

### Phase 0 — pre-flight

`mcp__github__get_me` returned `jonathanmanton`. `git remote -v` revealed the proxied repo is `lago-morph/pipeline-ai-sandbox` — a mismatch with the bundle's expectation (`<agent-account>/pipeline-ai-sandbox`). Python 3.11 was the only interpreter; the bootstrap install accepts 3.11 as a fallback for 3.12. The MCP scope was hard-limited to the current repo. This single fact dictated the shape of every subsequent live-execution phase.

### Phase 1 — initial scaffolding

Four subagents launched in parallel with disjoint write scopes (README; AGENTS+CLAUDE; project tooling; CI workflows). No worktree isolation — the path partition was obviously safe. 3 returned synchronously; sub-04 ran async and reported via webhook ~3 minutes later. All four landed without conflicts; the dispatcher committed the merged output as a single Phase 1 commit.

### Phase 2 — protocol self-install

`orchestrate-issue/templates/agent/*` is the superset of all three skills' self-install payloads. The dispatcher copied it wholesale into `.agent/` plus the three workflow YAMLs. The `_agent_runs` orphan branch was created locally and pushed.

**Operational mishap**: the orphan-branch creation used `git checkout --orphan _agent_runs && git rm -rf . && git clean -fdx`. Phase 2's freshly-copied `.agent/` and three workflow files were still untracked at that moment. `git clean -fdx` removed them along with the orphan-branch ancestors. After the round-trip back to the working branch, `git status` was clean — looked correct because the *tracked* files survived (git checkout restores them), but the *new* Phase 2 work was gone. Caught by an explicit `ls .agent/` after the round-trip; the Phase 2 copies were redone before commit. **The skill's documented orphan-branch procedure has this bug latent in it.**

The `agent-task` label could not be created — the github MCP server in this environment exposes `get_label` but no create-label tool. Documented as a deferred step.

### Phase 3 — archetype verification

Inline rather than fanout: each archetype's `manifest.json` vs on-disk listing took ~30ms; eight subagents would have been overhead, not parallelism. All 8 archetypes' manifests matched their files exactly. No drift.

### Phase 4 — scenario harness + runners

Discovered `test-harness/lib/*.py` was not in the bootstrap bundle (the SPEC.md listed it; the bundle omitted it). Wrote `state`, `archetype_loader`, `assertions`, `scenario_runner`, and a new module `synthetic_observe` from scratch. Then generated 18 runners from a single template + the 18 scenario YAMLs (a 30-line generator script).

**Operational mishap**: the bootstrap-shipped `.gitignore` had a `lib/` line (without leading slash), which matched `test-harness/lib/` and quietly excluded the entire just-written shared library from `git status`. The 18 runners showed up; the lib did not. Caught only because `git status --short` showed just `?? test-harness/runners/` — three files, not seven. Tightened to `/lib/` and `/lib64/` (only top-level virtualenvs).

### Phase 5 — sweep

First attempt: a Bash one-liner with `rm -rf harness/runs/$RUN_ID && for r in ...; do python3 "$r" >harness/runs/$RUN_ID/$sid.log 2>&1 || true; done`. The `rm` removed the parent directory; the `>` redirect failed for every iteration; `|| true` masked the failures; `$?` after `|| true` was always `0`. All 18 exit codes printed as 0; no runs actually executed. Caught by `find harness/runs -name state.json | wc -l` → 0.

Re-ran with `mkdir -p` first. 18/18 ran cleanly. 1 full pass (`composition-guide-render`), 4 partial (onboarding `detect` phases + `protocol-installed-not-onboarded`), 13 fully-skipped with `requires-live-skill-execution` diagnostic.

### Phase 6 — analysis

Aggregated `harness/runs/Vs1aL/*/state.json` into `runs/Vs1aL/test-results.md`. 0 failures, 6 of 72 phases done, 66 skipped. The 8% pass-rate is structural — most assertions reference live-execution artifacts.

## Part 4 — proposed ADRs

- **Synthetic-fixture is the default test target; live-new-repo is opt-in.** Grounded in: MCP scope is a hard floor on what `live-new-repo` can do here, and the friction of switching it requires explicit user action.
- **Bootstrap-installed paths are non-editable in this repo; changes round-trip through the POC source.** Grounded in: 27 of 29 ruff hits lived in those paths; treating them as editable would diverge them from the bundle rebuild.
- **Lint exemptions for vendored code go in `pyproject.toml`, not in `continue-on-error: true`.** Grounded in: the post-merge CI episode where "Success" hid 29 hits because the bypass was at the job level.
- **PRs open as draft; promote only when CI is genuinely green.** Grounded in: the user-prompted promotion after CI cleared, after the ruff fix exposed an early-promote bug.
- **Per-scenario harness state under `harness/runs/` is gitignored; dispatcher run state under `runs/` is committed.** Grounded in: per-scenario state is reproducible from runners; dispatcher state is the audit trail.
- **Skills in `.claude/skills/` are user-invocable via slash command and via the skill catalogue; SKILL.md frontmatter is the contract.** Grounded in: the post-merge commit `8663100` added 6 skills with no other plumbing — frontmatter alone was enough.

The user decides per ADR whether to author one via the `adr` skill. The retrospective lists titles only; no spec is provided per ADR.

## Part 5 — Correction (2026-05-16)

The Phase 7 dogfood-blocked finding ("MCP has no label-create tool, so the `agent-task` label could not be applied") was a **misdiagnosis** rooted in not consulting the PoC bundle's design. The PoC pattern is: the agent never applies the label — `lock-and-sweep.yml` does so automatically on `issues.opened` after verifying the issue creator is authorised. The agent's only job is to open the issue with a valid `agent-meta` block. If the label was missing on issue #1, the bug is "workflow didn't fire / didn't pass its `creator == agent_login` check", not "MCP toolset is too restrictive."

This is now codified in [`AGENTS.md` §6](../AGENTS.md) (GitHub MCP discipline). The follow-up PR generalised the `lock-and-sweep.yml` gate from a hardcoded `vars.AGENT_LOGIN || 'jonathanmanton'` literal to an `author_association`-based check (any user with repo write access can drive the protocol), with `vars.AGENT_LOGIN` retained as an optional single-bot pin.
