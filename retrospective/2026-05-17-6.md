### Phase 1 — probing questions exposed mis-framings

The user asked three sharp follow-ups, each of which made me walk back a previous claim:

1. **"Why does phase 5 need to create repos?"** — Forced me to actually read `test-harness/SPEC.md` §"Real-GitHub interaction model" and the scenario YAMLs. 8 scenarios target `live-new-repo`; 10 are `synthetic-fixture`. Live scenarios need real workflow firings (the handler workflow is what stamps terminal envelopes; no off-GitHub way to exercise it). A *fresh* repo per scenario is hermeticity, not safety.

2. **"Does your MCP server even have the ability to manipulate multiple repos at once?"** — This caught a real misframing in both my answer AND the prior retro. I had said "MCP scope is a hard floor". The MCP server fundamentally takes `owner` + `repo` as parameters and supports `create_repository` / `fork_repository`. The single-repo restriction is a **policy layer wrapped around this Claude-Code-on-the-Web session** — it's in the system prompt under "Repository Scope". My earlier framing pinned the wrong layer.

3. **"So if I ran this on a cloud code CLI in Linux..."** — Confirmed yes: a self-hosted CLI with a token whose scope permits repo creation is the environment the plan was originally written for.

### Phase 3 — build batch-job-happy-path live

Built four pieces:

- `test-harness/lib/envelopes.py` — build / serialize / parse `batch-job-request` envelopes. Lenient prefix-JSON parse mirrors `handler.py::_parse_envelope_lenient` so trailing prose (Claude Code's GitHub MCP appends a footer) doesn't break the round-trip.
- `test-harness/lib/live_observe.BatchJobObserver` — class with three phase methods (`setup` / `invoke` / `verify`) that drive a real repo and return observation dicts. Polling cadence, clock, and sleep injectable for deterministic tests. State held on `self.state` across phase calls.
- `test-harness/lib/github_client_factory.py` — resolves `GITHUB_TOKEN`/`GH_TOKEN` + `owner/repo` from env (or `origin` remote URL). Lazy-imports `RestGitHubClient` from `.agent/scripts/rest_client.py` via `importlib.util.spec_from_file_location` to avoid forking the protocol's REST client.
- Modified `test-harness/lib/scenario_runner.py` to do target negotiation: when the runner exposes a `live_observer_factory` AND creds resolve, use live; otherwise record `degraded_reason`. Removed the unconditional degradation.

## Part 4 — proposed ADRs

Architectural decisions made in this session that the user may want to record as ADRs. **Titles only — no specs.** Use the `adr` skill to author any of these if you decide they're worth recording.

- **Live-target negotiation lives in the scenario runner, not per-runner scripts.** `scenario_runner.run_scenario` accepts a `live_observer_factory` kwarg and decides target at runtime based on env detection; failure to build the live client records `degraded_reason` rather than crashing. Grounded in: removing the unconditional `target = "synthetic-fixture"` degradation in `scenario_runner.py`.
- **Test-harness lib imports `.agent/scripts/` modules via `importlib.util.spec_from_file_location`, not via package-relative imports.** Avoids forking the protocol's REST client + InMemoryGitHubClient while keeping the harness lib decoupled from `.agent` being a Python package. Grounded in: `github_client_factory.make_live_client` and `test_runner_batch_job_happy_path._load_common`.
- **Polling cadence configured per-observer, not from `.agent/config.json`.** `BatchJobObserver` takes `poll_interval_s` + `poll_timeout_s` as constructor kwargs (defaults 5s / 300s) rather than reading the protocol config. Reasoning: the dispatcher's polling is observer-side, not handler-side; tests need fast polling and config-coupling makes that harder. Grounded in: the test fixtures pass 0.0 / 10.0.
