# Spec: `runner-template-generation`

## Intent

When N near-identical artifacts are derived from a single canonical source (N scenarios → N runner scripts; N endpoints → N handler stubs; N tables → N migration files; N feature flags → N config keys), the productivity-vs-correctness trade-off is decisive: a single generator script + the canonical source is cheaper to write, cheaper to maintain, and structurally correct. Handcrafting each artifact, or dispatching N subagents one-per-artifact, multiplies effort and creates drift.

In the `pipeline-ai-sandbox` run, the original plan (`NEW-REPO-PLAN.md` Phase 4) called for 5 waves of 4 subagents each (20 dispatches) to write 18 scenario runners. The dispatcher instead wrote a 30-line generator that read the 18 scenario YAMLs and emitted 18 runner files. The result: less work, less drift, easier-to-update artifacts (re-running the generator is the canonical update).

## Trigger

**Direct triggers:**
- "Write N <thing> from this list / spec / catalogue."
- "Generate stubs for each <X>."
- "We have a single source-of-truth — produce one <Y> per entry."

**Proactive triggers:**
- A task brief lists N artifacts where N > 3 and the structure of each is nearly identical.
- A plan calls for "M waves of K subagents" to produce N similar files.
- An existing repo has N files in a directory whose names match a pattern, and they're visibly drifting (different copyright headers, different formatter applied, different import order).

**Negative triggers:**
- The N artifacts are genuinely independent (different problem domains, different libraries, different schemas).
- N ≤ 2 — a generator is overkill.
- The artifacts are one-shot scaffolding that will be heavily edited afterwards (the generator overhead exceeds the saved time).

## Inputs

- The canonical source: a YAML / JSON / TOML / database / spec file enumerating N entries.
- The target output directory.
- A template for the artifact, parameterised by per-entry values.
- (Optional) a hook for per-entry specialisation when the template alone isn't enough.

## Outputs

- A generator script committed in the repo (e.g., `scripts/generate_runners.py`).
- N artifact files at the target path.
- (Optional) a generation marker comment at the top of each generated file: "Generated by `scripts/generate_runners.py`. Do not edit by hand; regenerate."
- (Optional) a CI check that runs the generator and fails if the working tree changes.

## Workflow

1. **Locate the canonical source.** If it doesn't exist, write it first — the spec IS the source of truth.
2. **Choose a template.** Prefer a string-format / Jinja2 / f-string template inside the generator script for ≤ 100 lines per artifact. Externalise to `<artifact>.template` only when the template grows complex.
3. **Write the generator.** Idempotent: re-running with the same source produces byte-identical artifacts.
   ```python
   for entry in canonical_source:
       artifact = TEMPLATE.format(**entry_to_dict(entry))
       (out_dir / f"{entry.id}.py").write_text(artifact)
   ```
4. **Validate the output.** Compile / lint / parse each generated artifact. The generator is correct only if every output is valid.
5. **Commit both the generator and the outputs.** The outputs are committed too (developers shouldn't need to run the generator just to read the code).
6. **Optional: CI guard.** Add a CI step that runs the generator and `git diff --exit-code` to ensure the outputs match. Forces re-generation if the source changes without the artifacts being refreshed.
7. **Document the contract.** A line in `AGENTS.md`: "Files under `<out_dir>` are generated by `<script>`. Edit `<source>` and re-run, don't edit the outputs."

## Concrete examples

### Example 1 — `pipeline-ai-sandbox` (this session)

Canonical source: 18 scenario YAML files at `test-harness/scenarios/*.yml`. Target: 18 runner files at `test-harness/runners/*.py`. Template inside the generator:

```python
runner_template = '''#!/usr/bin/env python3
"""Runner for the {sid!r} scenario.
...
"""
from __future__ import annotations
import sys
from pathlib import Path

_LIB = Path(__file__).resolve().parents[1] / "lib"
if str(_LIB) not in sys.path:
    sys.path.insert(0, str(_LIB))

from scenario_runner import cli_main
from synthetic_observe import generic_observe

SCENARIO_ID = {sid!r}
PHASE_EXPECTED_KEYS = {phase_keys!r}

def observe(phase_name, inputs, fixture, diagnostics):
    return generic_observe(phase_name, inputs, fixture, diagnostics,
                           expected_keys=PHASE_EXPECTED_KEYS.get(phase_name, []))

if __name__ == "__main__":
    raise SystemExit(cli_main(SCENARIO_ID, observe))
'''

for sid, scenario in scenarios.items():
    phase_keys = {ph['name']: sorted((ph.get('expected') or {}).keys()) for ph in scenario['phases']}
    out = Path('test-harness/runners') / f'{sid}.py'
    out.write_text(runner_template.format(sid=sid, phase_keys=phase_keys))
```

Result: 18 runners, all compilable, all uniform. Adding a new scenario is now "add the YAML; re-run the generator." The 18 runners share zero hand-edited drift.

### Example 2 — N migration stubs from a schema diff

Canonical source: a `schema-diff.json` listing tables added/changed/dropped between version M and N. Target: one Alembic migration file per change.

```python
TEMPLATE = '''"""{description}

Revision ID: {rev}
Revises: {down_rev}
Create Date: {created_at}
"""
from alembic import op
import sqlalchemy as sa

revision = '{rev}'
down_revision = '{down_rev}'

def upgrade():
{upgrade_body}

def downgrade():
{downgrade_body}
'''

for change in schema_diff["changes"]:
    body = render_alembic_ops(change)  # per-change specialisation
    (migrations_dir / f"{change.rev}_{change.slug}.py").write_text(
        TEMPLATE.format(**change.as_template_vars(), **body)
    )
```

Hand-editing the migrations diverges them from the diff; the generator + the diff are the source of truth.

## Anti-patterns

- **N subagents for N near-identical artifacts.** The classic plan-document mistake: "5 waves of 4 subagents to write 18 files." One generator is faster and produces uniform output.
- **Generator output that isn't byte-identical on re-run.** Sort dict keys, fix newline style, freeze the formatter version. Non-determinism makes the CI guard useless.
- **Hand-editing generated files.** They get overwritten on the next regen. If a per-entry specialisation is needed, lift it into the template or into a per-entry hook in the generator.
- **No "generated by" comment at the top of outputs.** Future readers will edit by hand because they don't know the file is generated.
- **Generator script committed but generated outputs not committed.** Forces every reader to run the generator just to see the code. Commit both.
- **No CI guard.** Source changes silently get out of sync with outputs.

## Acceptance criteria

1. Running the generator twice produces byte-identical outputs (idempotent).
2. Every output is syntactically valid (compiles / lints / parses).
3. The canonical source is the single point of edit; outputs are regenerated.
4. A CI step verifies the outputs match the source (`generator && git diff --exit-code`).
5. Each generated file has a top-of-file marker identifying it as generated.
6. The generator and the outputs are both committed.

## Files this skill creates / modifies

- A generator script (typically `scripts/generate_<thing>.py` or `tools/codegen/<thing>.py`).
- N generated artifact files at the target path.
- A line in `AGENTS.md` / `CONTRIBUTING.md` naming the generator as the source-of-truth for the outputs.
- Optionally a `.github/workflows/generated-up-to-date.yml` (or equivalent) CI check.
